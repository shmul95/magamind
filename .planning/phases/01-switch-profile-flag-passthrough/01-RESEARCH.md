# Phase 1: switch-profile flag passthrough - Research

**Researched:** 2026-05-07
**Domain:** Bash argument parsing, Home Manager activation internals
**Confidence:** HIGH

## Summary

This phase adds flag support to `switch-profile.sh` so users can pass Home Manager activation options (`-b`, `-n`, `--show-trace`) when switching profiles. The script currently takes exactly one argument (profile name) and passes nothing to `activate`.

**Critical finding:** The CONTEXT.md decision D-03 ("blind passthrough — forward all args except the last to `activate`") is **incompatible with the actual `activate` script**. The Home Manager-generated `activate` script has a strict argument parser that **only** accepts `--driver-version N` and rejects all other flags with an error (`unknown option '%s'`). The flags `-b`, `-n`, `--show-trace` are handled by the `home-manager` CLI wrapper, which translates them into **environment variables** (`HOME_MANAGER_BACKUP_EXT`, `DRY_RUN`, `VERBOSE`) and calls `activate` with only `--driver-version 1`. [VERIFIED: inspected `/home/shmul95/.local/share/cabanashmul/result-personal/activate` lines 18-39 and `home-manager` CLI source lines 1170-1290]

**Primary recommendation:** Replace blind passthrough with **flag-to-env-var translation**. The script must parse known flags, export the corresponding environment variables, and call `activate` with `--driver-version 1` only. This matches how `home-manager switch` itself works.

## Critical Finding: D-03 Blind Passthrough Is Incorrect

**Evidence chain:**

1. The `activate` script at `result-<profile>/activate` is generated by Home Manager's Nix code [VERIFIED: inspected live script]
2. Its argument parser (lines 18-39) accepts **only** `--driver-version N` and exits with an error on any other flag [VERIFIED: inspected live script]
3. The `home-manager` CLI (the canonical caller of `activate`) handles `-b`, `-n`, `-v`, `--show-trace` etc. itself and **never passes them to activate** [VERIFIED: inspected `home-manager` CLI source lines 815-836]
4. Instead, `home-manager` exports environment variables that `activate` reads:
   - `-b EXT` → `export HOME_MANAGER_BACKUP_EXT="$EXT"` [VERIFIED: CLI line 1181]
   - `-n` → `export DRY_RUN=1` [VERIFIED: CLI line 1229, activate line 198]
   - `-v` → `export VERBOSE=1` [VERIFIED: CLI line 1283, activate line 172]
   - `--show-trace` → added to `PASSTHROUGH_OPTS` for nix-build, plus `VERBOSE=1` also triggers `--show-trace` on internal nix calls [VERIFIED: CLI line 263-264, 575-576]

**Impact on D-03:** Blind passthrough will cause `activate` to print `unknown option '-b'` and exit 1. The script **must** translate flags to environment variables.

**Recommended resolution:** Replace D-03 with **selective parsing** of the three flags in scope (`-b`, `-n`, `--show-trace`), plus a pass-unknown approach for any remaining args (which will naturally error from `activate`, satisfying D-07). This preserves D-01 (last arg = profile), D-02 (no separator), and D-05 (updated usage line).

## Architecture Patterns

### System Architecture Diagram

```
User invocation
       │
       ▼
switch-profile [FLAGS...] <profile-name>
       │
       ├─ Parse flags: -b EXT, -n, --show-trace
```

### Pattern 1: Flag-to-Environment Translation

**What:** Parse CLI flags in the wrapper script, export environment variables, call activate with `--driver-version 1` only.
**When to use:** Any time wrapping Home Manager's activate script outside the `home-manager` CLI.

```bash
# Source: verified from home-manager CLI source (lines 1170-1290)
# and activate script (lines 172-206)

ACTIVATE_FLAGS=()
while (( $# > 1 )); do
  case "$1" in
    -b)
      shift
      export HOME_MANAGER_BACKUP_EXT="$1"
      ;;
    -n|--dry-run)
      export DRY_RUN=1
      ;;
    --show-trace)
      export VERBOSE=1
      ;;
    *)
      ACTIVATE_FLAGS+=("$1")
      ;;
  esac
  shift
done
PROFILE_NAME="${1:-}"
```

### Pattern 2: Last-Arg Extraction (Bash)

**What:** Extract the last positional argument as the profile name; collect all preceding args.
**When to use:** CLIs where the target/subject is the final positional arg.

```bash
# Idiomatic bash approach — works with zero or many flags
if (( $# == 0 )); then
  echo "Usage: switch-profile [FLAGS...] <profile-name>" >&2
  exit 1
fi

# Process all args except last
while (( $# > 1 )); do
  # handle "$1"
  shift
done
PROFILE_NAME="$1"
```

### Anti-Patterns to Avoid
- **Blind passthrough to activate:** Passing arbitrary CLI flags to activate will cause it to error on any flag except `--driver-version`. [VERIFIED]
- **Using `getopts` for long options:** Bash's built-in `getopts` does not support long options (`--show-trace`). Use a `while/case` loop instead. [ASSUMED — standard bash knowledge]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Flag parsing with long opts | Custom `getopt` wrapper | `while/case` loop | Simple, no external dependency, handles both short and long flags |

**Key insight:** The flag set is small (3 flags) and well-defined. A `while/case` loop is the simplest correct approach.

## Common Pitfalls

### Pitfall 1: Blind Passthrough to activate
**What goes wrong:** `activate` prints `unknown option '-b'` and exits 1.
**Why it happens:** Assumption that `activate` accepts the same flags as `home-manager switch`.
**How to avoid:** Translate flags to environment variables as `home-manager` does.
**Warning signs:** Any `activate` call with args other than `--driver-version N`.

### Pitfall 2: Missing `shift` After `-b`
**What goes wrong:** The backup extension value gets treated as the profile name.
**Why it happens:** `-b` takes a required argument (`-b backup`), unlike `-n` which is standalone.
**How to avoid:** Always `shift` after consuming `-b`'s argument.
**Warning signs:** Profile name resolves to "backup" instead of the actual profile.

### Pitfall 3: `--show-trace` vs `VERBOSE`
**What goes wrong:** User expects `--show-trace` to only add trace output, but `VERBOSE=1` also enables verbose echo throughout activation.
**Why it happens:** Home Manager conflates `--show-trace` with `VERBOSE` in the activate script — the `home-manager` CLI does the same (line 263-264: when VERBOSE is set, `--show-trace` is added to nix args).
**How to avoid:** Accept this as the intended behavior — it matches `home-manager -v` semantics. Document that `--show-trace` implies verbose output.
**Warning signs:** User confused by extra output when they only wanted traces.

### Pitfall 4: Empty `$#` After Flag Consumption
**What goes wrong:** All args consumed by flag parsing, no profile name left → `PROFILE_NAME` is empty.
**Why it happens:** User runs `switch-profile -n` without specifying a profile.
**How to avoid:** After flag parsing, check that exactly one arg remains (the profile name). If not, print usage and exit 1.
**Warning signs:** Empty PROFILE_NAME or `unbound variable` error from `set -u`.

### Pitfall 5: `--driver-version 1` Missing
**What goes wrong:** Activate script runs with `hmDriverVersion=0` (legacy behavior) which may skip the profile migration.
**Why it happens:** Current script calls `exec "$RESULT_LINK/activate"` without `--driver-version 1`.
**How to avoid:** Add `--driver-version 1` to the `exec` call, matching how `home-manager` invokes activate.
**Warning signs:** Profile not properly tracked in nix-env profiles after activation.

## Code Examples

### Complete Implementation Pattern

```bash
#!/usr/bin/env bash
set -euo pipefail

# --- Flag parsing ---
if (( $# == 0 )); then
  echo "Usage: switch-profile [FLAGS...] <profile-name>" >&2
  exit 1
fi

while (( $# > 1 )); do
  case "$1" in
    -b)
      shift
      if (( $# < 2 )); then
        echo "switch-profile: -b requires an argument" >&2
        exit 1
      fi
      export HOME_MANAGER_BACKUP_EXT="$1"
      ;;
    -n|--dry-run)
      export DRY_RUN=1
      ;;
    --show-trace)
      export VERBOSE=1
      ;;
    *)
      echo "switch-profile: unknown flag '$1'" >&2
      exit 1
      ;;
  esac
  shift
done

PROFILE_NAME="$1"

# --- Profile resolution (unchanged) ---
DATA_DIR="${XDG_DATA_HOME:-$HOME/.local/share}/cabanashmul"
RESULT_LINK="$DATA_DIR/result-$PROFILE_NAME"

if [ ! -e "$RESULT_LINK" ]; then
  echo "switch-profile: no pre-built result for '$PROFILE_NAME'" >&2
  echo "Run build-profiles from your cabanashmul directory first." >&2
  exit 1
fi

echo "Activating profile: $PROFILE_NAME"
exec "$RESULT_LINK/activate" --driver-version 1
```

Source: Pattern derived from `home-manager` CLI source (verified) and project conventions (CONVENTIONS.md).

## Open Questions

1. **D-03 conflict resolution**
   - What we know: Blind passthrough will break. Flag-to-env translation is required.
   - What's unclear: Whether the user wants to keep the "forward unknown flags" behavior (which will error) or reject unknowns early with a helpful message.
   - Recommendation: Reject unknown flags with `switch-profile: unknown flag '<flag>'` — this gives a better error than the cryptic `activate` error. Flag this to user in planning/discuss.

2. **Should `--driver-version 1` be added?**
   - What we know: Current script omits it; `home-manager` always passes it. Running without it uses legacy behavior (version 0).
   - What's unclear: Whether the omission was intentional or an oversight.
   - Recommendation: Add `--driver-version 1` to match `home-manager switch` behavior. This is a correctness improvement.

3. **`--show-trace` semantics**
   - What we know: The only way to get `--show-trace` on internal nix calls during activation is `export VERBOSE=1`, which also enables verbose echo output.
   - What's unclear: Whether the user expects `--show-trace` to ONLY add traces (no extra verbosity).
   - Recommendation: Accept the coupling — it matches `home-manager -v` behavior, and the extra verbosity is helpful context alongside traces.
